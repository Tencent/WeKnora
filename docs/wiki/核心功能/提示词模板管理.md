# 提示词模板管理

## 概述

提示词模板（Prompt Templates）是 WeKnora 给到所有 Agent / 普通会话使用的"提示词候选库"，对应前端 Agent 编辑器里的"模板下拉"。

历史上模板内容只能写在 `config/prompt_templates/*.yaml` 里，**改一次就要重启服务**。从这一版起，**5 类用户面向的模板被纳入数据库管理**，配合一个写接口可以做到：

- 改完即时生效，不用重启
- YAML 仍然是"出厂默认"，新部署或新增模板会被自动 seed 进库
- 改坏了可以通过 PUT 把 YAML 默认值贴回去恢复

> 同为系统级配置的还有 [[内置模型管理]] 和 [[内置MCP服务管理]]。

---

## 涵盖范围（哪 5 类进了 DB）

只有以下 5 个分类进了 `prompt_templates` 表，它们恰好是前端 `PromptTemplateSelector` 里能选到、并被复制进 Agent 配置的那批：

| 分类（category） | 用途 | 谁会读它 |
|---|---|---|
| `system_prompt` | 普通模式 system prompt | Agent 编辑器、普通会话默认 |
| `agent_system_prompt` | 智能体模式 system prompt | Agent 编辑器（智能体类型） |
| `context_template` | 检索结果如何拼进上下文 | Agent 编辑器、普通会话默认 |
| `rewrite` | 用户问题改写（含双 prompt） | Agent 编辑器 |
| `fallback` | 没有匹配时的兜底文案 | Agent 编辑器 |

**不在 DB、仍然只读 YAML 的 6 类**：

`generate_session_title` / `generate_summary` / `generate_questions` / `keywords_extraction` / `graph_extraction` / `intent_prompts`

这些是开发者维护的"全局服务能力"模板，普通运营/管理员不会动，改它们仍然需要修 YAML 后重启。

## 数据流

```
┌─────────────────────────────┐
│ config/prompt_templates/*.yaml │   ← 出厂默认（随代码版本演进）
└──────────────┬──────────────┘
               │ 启动时 LoadConfig 读取
               ▼
       cfg.PromptTemplates
               │
               │ 启动时 SeedAndLoadPromptTemplates
               │   ① YAML 中存在但 DB 缺失 → INSERT
               │   ② 读全表 → 覆盖 cfg.PromptTemplates 的 5 个分类
               ▼
┌─────────────────────────────┐
│      prompt_templates 表     │   ← 运行时唯一来源
└──────────────┬──────────────┘
               │
               │ 改动后 RefreshPromptTemplatesFromDB
               │   ① 重读全表
               │   ② 刷新 cfg.Conversation.* 中按 ID 关联的字段
               │   ③ 重新 resolve 内置 Agent 的 SystemPrompt 缓存
               ▼
        前端 / Chat Pipeline
```

### 启动时的两种场景

| 场景 | 行为 |
|---|---|
| **首次部署 / 表为空** | seed 步骤把 YAML 全部内容写入 DB；之后 cfg = DB 内容（等价于 YAML） |
| **已运行过 / 表有数据** | seed 步骤只补齐 YAML 里的"新模板"（DB 里不存在的 (category, id)）；用户已修改的行不动 |

### 删除 / 缺失的语义

- 当前 API 不再提供 DELETE 接口（小节"修改方式"会说明原因）
- 如果通过 SQL 直接删掉一行：cfg.PromptTemplates 里那条会消失；下次重启 seed 会从 YAML 把它再插回来（实质等于"恢复出厂"）

## 修改方式 1：调用接口（推荐）

### 接口列表

| 方法 | 路径 | 鉴权 | 用途 |
|---|---|---|---|
| `GET` | `/api/v1/tenants/kv/prompt-templates` | 任意已登录租户 | 拉取全部模板（按当前语言本地化） |
| `PUT` | `/api/v1/tenants/kv/prompt-templates` | **仅超级管理员** | 新增 / 更新单条模板 |

> 没有 DELETE：模板表是慢变、低基数的配置，删除场景几乎不存在；要"恢复出厂"用 PUT 把 YAML 默认值贴回去即可。

### 谁是超级管理员

需同时满足：

1. `users.can_access_all_tenants = true`（DB 字段，目前没有 UI/API 能改）
2. 该用户处于已登录态（JWT）或其租户的 API Key 间接映射到该用户

通过 SQL 提升一次：

```sql
-- PostgreSQL
UPDATE users SET can_access_all_tenants = TRUE WHERE email = 'you@example.com';

-- SQLite
UPDATE users SET can_access_all_tenants = 1   WHERE email = 'you@example.com';
```

提升后**重新登录**拿新 JWT，或确认你用的 API Key 对应的就是该用户（`X-API-Key` 鉴权时中间件会取所属租户的第一个用户）。

> 鉴权设计上**没有**绑 `cfg.Tenant.EnableCrossTenantAccess`，因为 prompt_templates 是全局资源而非"另一个租户的资源"。

### PUT 请求字段

| 字段 | 必填 | 说明 |
|---|---|---|
| `category` | ✅ | 5 类之一 |
| `id` | ✅ | 在该 category 下唯一，长度 ≤ 64 |
| `content` | ✅ | 主体内容，长度 ≤ 32K |
| `name` | ❌ | 显示名（255） |
| `description` | ❌ | 简介（1024） |
| `user` | ❌ | 仅 `rewrite` 用：用户侧子提示词 |
| `has_knowledge_base` | ❌ | 是否依赖知识库（UI 标记） |
| `has_web_search` | ❌ | 是否依赖网络搜索（UI 标记） |
| `default` | ❌ | 是否为该 category 的默认项 |
| `mode` | ❌ | 仅 `fallback` 用：`""` 固定回复 / `"model"` 模型生成 |
| `i18n` | ❌ | `{ "zh-CN": {name, description}, "en-US": {…} }` |

### curl 示例

更新 `system_prompt/default_kb`：

```bash
curl -sS -X PUT "http://localhost:8080/api/v1/tenants/kv/prompt-templates" \
  -H "X-API-Key: $WEKNORA_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "category": "system_prompt",
    "id": "default_kb",
    "name": "Knowledge Base Q&A",
    "content": "你是 caiwutong，一个严谨的产品文档助手。\n\n上下文：{{contexts}}\n\n问题：{{question}}",
    "has_knowledge_base": true,
    "default": true
  }' | jq .
```

新建一条供选择的自定义模板：

```bash
curl -sS -X PUT "http://localhost:8080/api/v1/tenants/kv/prompt-templates" \
  -H "X-API-Key: $WEKNORA_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "category": "system_prompt",
    "id": "my_custom_kb",
    "name": "我的定制助手",
    "description": "用于产品文档问答",
    "content": "...",
    "has_knowledge_base": true
  }' | jq .
```

> 鉴权方式可任选 `X-API-Key: $WEKNORA_API_KEY` 或 `Authorization: Bearer $WEKNORA_TOKEN`，效果等价。

## 修改方式 2：直接改库

直接 `UPDATE prompt_templates SET content = '...' WHERE category = 'system_prompt' AND id = 'default_kb';` 也是可行的，但要注意：

- **当前实现内存有缓存**：cfg.PromptTemplates 在启动时灌一次，**只在 PUT 接口被调用时**会触发刷新。直接改库 → **必须重启服务**才能生效。
- 接口 PUT 同一条记录 1 次（即使内容相同）就会触发一次 `RefreshPromptTemplatesFromDB`，等同于"无重启刷新"。

## 内置 Agent 的同步

四个内置 Agent（快速问答 / 数据分析师 / 维基问答 / 维基修订）在 YAML 里只写 `system_prompt_id`，启动时会**把模板内容拷贝到内置 Agent 的 SystemPrompt 字段**作缓存。

每次 PUT 接口写库后，会做：

1. `ResetBuiltinAgentResolvedPrompts()`：清空那 4 个内置 Agent 中"由 ID 解析出的"SystemPrompt / ContextTemplate
2. `ResolveBuiltinAgentPromptRefs()`：用最新 cfg.PromptTemplates 重新填回去

这样 `GET /api/v1/agents/builtin-quick-answer` 立刻能返回新内容；前端"快速问答"里看到的系统提示词也会跟着变。

> 智能推理（`builtin-smart-reasoning`）在 YAML 里 `system_prompt: ""`，运行时通过 `agent_type=rag-qa` → `agent_type_presets.yaml` → `system_prompt_id=progressive_rag_agent` 由前端实时查模板，**自然同步**，无需特殊处理。

---

## 用户已创建 Agent 不会同步

每个用户自建 Agent（`custom_agents` 表）保存时，前端会把"当前选中模板"的内容**复制**到 `config.system_prompt`。这是产品语义：

- Agent 一旦创建，它的提示词就是"它自己的"
- 之后修改模板表**不会**反向影响已存在的 Agent
- 这避免了 SaaS 场景下管理员改模板"无声无息地改了所有用户 Agent 行为"的副作用

如果要让某个 Agent 重新同步到模板最新值：

- 前端：进入 Agent 编辑器 → 点"重置为默认"
- SQL：`UPDATE custom_agents SET config = jsonb_set(config, '{system_prompt}', '""'::jsonb) WHERE id = '...';`
  清空之后，运行时回退到 `cfg.Conversation.Summary.Prompt`（来自模板，已是最新）

## 常见问题

### Q1：改完模板，前端"快速问答"的系统提示词没变？

**自查清单**：

```bash
# ① API 是否真的拿到了新内容
curl -sS "http://localhost:8080/api/v1/tenants/kv/prompt-templates" \
  -H "X-API-Key: $WEKNORA_API_KEY" \
  | jq '.data.system_prompt[] | select(.id=="default_kb") | .content'

# ② 内置 Agent 的缓存是否同步
curl -sS "http://localhost:8080/api/v1/agents/builtin-quick-answer" \
  -H "X-API-Key: $WEKNORA_API_KEY" \
  | jq -r '.data.config.system_prompt' | head -2
```

- ① 返回的就是新的 → DB / cfg 这一层没问题
- ② 返回旧的 → 你是直接 SQL 改的、没走 PUT 接口；改用 PUT 或重启

### Q2：用 X-API-Key 调 PUT 报 403？

中间件会把 API Key 的"租户"映射到"该租户的第一个用户"。要那个用户的 `can_access_all_tenants = true`：

```sql
SELECT id, email, can_access_all_tenants FROM users WHERE tenant_id = <your_tenant_id> ORDER BY created_at LIMIT 1;
-- 把上面查出来的那个用户提升为超管
UPDATE users SET can_access_all_tenants = TRUE WHERE id = '<above_id>';
```

### Q3：误改了模板想恢复出厂？

PUT 把 YAML 默认值贴回去：

```bash
DEFAULT=$(yq '.templates[] | select(.id=="default_kb") | .content' \
  config/prompt_templates/system_prompt.yaml)

curl -sS -X PUT ".../tenants/kv/prompt-templates" \
  -H "X-API-Key: $WEKNORA_API_KEY" \
  -H "Content-Type: application/json" \
  -d "{
    \"category\":\"system_prompt\",
    \"id\":\"default_kb\",
    \"content\": $(echo \"$DEFAULT\" | jq -Rs .),
    \"has_knowledge_base\": true,
    \"default\": true
  }"
```

或者：直接 `DELETE FROM prompt_templates WHERE category='system_prompt' AND id='default_kb';` + 重启服务（启动 seed 会再插回来）。

### Q4：要不要把不在 DB 的那 6 类也接进来？

短期没必要：那 6 类是开发者维护的全局服务能力，运营/管理员不会动。如果真有需求，扩展点已经留好——加 CHECK 枚举、加 seed 列表、加 merge 字段、再补几处衍生字段刷新即可，~120 行内的事。
